第一层 · 传输基础
▶ 传输层 Linux 系统编程与网络基础 所有上层通信的根基

要理解什么

  • ▸ 进程间通信（IPC） — 管道、共享内存、信号、消息队列 → 为什么机器人软件不直接用这些？
  • ▸ Socket 编程 — TCP 可靠传输 vs UDP 低延迟 → DDS 底层用 UDP 多播，gRPC 底层用 TCP
  • ▸ 多播（Multicast） — UDP 多播是一对多高效分发的基础 → DDS 的发现协议和 Topics 依赖它
  • ▸ 序列化 — 把内存中的结构体变成字节流传输 → Protobuf / CDR 是两种不同的序列化方案
  • ▸ CMake 构建体系 — 管理多文件、多依赖的 C++ 编译 → SDK 和 ROS2 包都用 CMake
这一层解决什么问题：DDS 和 gRPC 不凭空出现，它们分别基于 UDP 多播和 TCP Socket。不理解 Socket 和序列化，就无法理解 DDS 的 QoS 和 gRPC 的 Stub 是怎么工作的。
第二层 · 中间件
▶ 中间件 DDS — 数据分发服务 SDK 底层 · 传感器数据流

核心概念

  • ▸ 发布-订阅模型（Pub-Sub） — 数据生产者和消费者解耦，谁也不需要知道对方的存在 → 和请求-响应模式的本质区别
  • ▸ Topic — 数据通道，按主题名订阅，类似"频道" → RGB图像、IMU、电机状态各自是不同 Topic
  • ▸ Domain & DomainParticipant — DDS 的隔离域，同一 Domain 下的节点才能互相通信
  • ▸ QoS（Quality of Service） — 可靠性、历史深度、传输策略等 → 传感器数据要"最新帧"还是"不丢帧"？QoS 决定
  • ▸ IDL（接口定义语言） — 定义数据结构 → SDK 中 RGB图像、IMU数据的消息格式就是这样定义的
  • ▸ 自动发现（Discovery） — 节点上线自动互相发现，无需中心注册 → 这就是为什么 DDS 是去中心化的

关键实现

  • ▸ Cyclone DDS — Eclipse 基金会开源，轻量级 → 你的 Dobot SDK 用的就是这个
  • ▸ Fast DDS（eProsima） — ROS2 Humble 默认实现，性能更优
  • ▸ 两者关系 — 都遵循 OMG DDS 规范，API 行为一致，底层实现和调优策略不同
这一层解决什么问题：传感器每秒产生大量数据（图像 30fps、IMU 200Hz），需要一种"发完就忘"的机制让多个消费者同时拿到数据。DDS 的 Pub-Sub + QoS 就是为此设计。

▶ 中间件 gRPC — 远程过程调用 SDK 高层 · 指令控制

核心概念

  • ▸ 请求-响应模型（Request-Response） — 客户端发请求，服务端返回结果 → 和 Pub-Sub 的"发完就忘"完全不同
  • ▸ Protobuf（Protocol Buffers） — Google 的序列化协议，定义 .proto 文件自动生成代码 → 比 JSON 更小更快，但不可读
  • ▸ Stub（桩代码） — 自动生成的客户端代理，调方法像调本地函数 → SDK 中 RobotClient 就是 gRPC Stub 的封装
  • ▸ 四种通信模式 — 一元 RPC、服务端流、客户端流、双向流 → SDK 高层控制主要用一元 RPC
  • ▸ HTTP/2 底层 — 多路复用、头部压缩、双向流 → gRPC 高性能的来源
这一层解决什么问题："让机器人站起来""走直线""切换状态"这类指令是偶发的、需要确认结果的。gRPC 的请求-响应模式天然适合：发了指令，等机器人确认执行完。

▶ 对比 DDS vs gRPC — 为什么 SDK 需要两个？ 理解选型背后的权衡
场景对比： ┌──────────────────────────────────────────────────────────────┐ │ 传感器数据流 (DDS) │ │ 相机 ──publish──▶ Topic: /rgb_image │ │ 相机 ──publish──▶ Topic: /depth_image │ │ IMU ──publish──▶ Topic: /imu_data │ │ │ │ 特点：高频、连续、多个消费者、不关心谁在听 │ │ 策略：发最新的就好，丢帧可以接受（QoS: best-effort） │ └──────────────────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────────────────┐ │ 指令控制 (gRPC) │ │ Client ──request──▶ "切换到 STAND 状态" ──response──▶ OK │ │ Client ──request──▶ "以 0.5m/s 前进" ──response──▶ OK │ │ │ │ 特点：低频、离散、需要确认结果、一对一 │ │ 策略：必须确认执行成功（可靠性优先） │ └──────────────────────────────────────────────────────────────┘
选型本质：DDS 擅长"数据洪流"（一对多、高频率、可丢帧），gRPC 擅长"精准指令"（一对一、需要确认、低频率）。机器人系统同时需要这两种通信模式，所以 SDK 分了两层。
第三层 · ROS2 框架
▶ 框架层 ROS2 架构本质 — DDS 的上层封装 理解 ROS2 不神秘

核心映射：ROS2 概念 → DDS 底层

  • ▸ Node（节点） → 一个 DomainParticipant，即一个独立的计算单元 → 一个进程可以包含多个节点
  • ▸ Topic（话题） → DDS Topic + Publisher/Subscriber → 和 SDK 底层的 Topic 是同一回事
  • ▸ Service（服务） → DDS 上的请求-响应封装 → 类似 gRPC，但走 DDS 通道
  • ▸ Action（动作） → 长时间任务：带反馈的请求-响应 → "导航到某点"就是 Action
  • ▸ Parameter（参数） → 节点运行时可调的配置项 → 调参不用改代码

关键设计

  • ▸ rmw（ROS Middleware）抽象层 — ROS2 不绑定特定 DDS 实现 → Cyclone DDS / Fast DDS / Connext 可插拔切换
  • ▸ Lifecycle Node（生命周期节点） — 节点有状态机：Unconfigured → Inactive → Active → Nav2 大量使用，启动顺序可控
  • ▸ Launch 系统 — 声明式启动多节点、配置参数 → 替代一堆 shell 脚本
  • ▸ TF（坐标变换） — 维护坐标系树，任何时刻可查询任意两坐标系间的变换 → 相机→机器人→世界，自动串联
这一层解决什么问题：裸 DDS 需要手动管理 Participant、Topic、QoS，很繁琐。ROS2 提供标准化的节点抽象、消息定义、工具链，让你专注于算法而不是通信细节。它不是魔法，就是 DDS 上面的一层工程封装。
ROS2 Design — 架构设计决策文档（为什么这样设计） ROS2 Humble 官方文档
▶ 框架层 仿真环境 — Gazebo 没有实体机器人也能开发
  • ▸ 物理引擎 — 模拟重力、碰撞、摩擦力，让虚拟机器人行为接近真实
  • ▸ 传感器仿真 — 生成虚拟的 RGB/深度图像、激光雷达、IMU 数据 → 数据格式和真实传感器一致
  • ▸ URDF/SDF 模型 — 用 XML 描述机器人的连杆、关节、传感器安装位置 → SDK resources/ 下有现成的 URDF
  • ▸ ROS2 集成 — Gazebo 里的虚拟传感器直接发布 ROS2 Topic → 代码不用改，仿真和真机切换
为什么重要：在真机上调试 SLAM 和导航很慢且有风险。Gazebo 让你在仿真里验证算法，通过后再部署到四足上。
第四层 · 感知与建图
▶ 感知层 传感器融合 — 多源数据怎么合到一起 建图的原料准备
  • ▸ 相机内参标定 — 焦距、畸变系数，把像素坐标映射到真实角度 → 深度图转换点云的基础
  • ▸ RGB-D 对齐 — 深度图和彩色图对齐，知道每个像素的距离 → SDK e1 + e2 两个 Topic 需要融合
  • ▸ IMU 预积分 — 加速度计 + 陀螺仪，提供两帧图像之间的运动约束 → 视觉的补充：快速运动时视觉会跟丢
  • ▸ 轮式里程计（轮足构型）— 电机编码器推算位移 → SDK e5 电机状态可算里程计
  • ▸ 时间同步 — 不同传感器频率不同，需要按时间戳对齐 → ROS2 message_filters 做近似/精确同步
这一层解决什么问题：单一传感器不够用——相机会被遮挡、IMU 会漂移、轮子会打滑。融合多个传感器取长补短，才能得到可靠的位姿估计。
▶ 建图层 SLAM — 同时定位与建图 "我在哪" + "周围长什么样"

SLAM 在解什么数学问题

  • ▸ 状态估计 — 同时估计机器人位姿（在哪里、朝向哪）和地图（环境结构）
  • ▸ 前端（数据关联） — 提取特征、匹配相邻帧，建立"我移动了多少"的初步估计 → 视觉特征点 ORB/SIFT，或激光扫描匹配
  • ▸ 后端（优化） — 把所有观测约束放到一起，解一个全局优化问题 → 图优化（g2o/Ceres）或滤波（EKF）
  • ▸ 回环检测 — "这个地方我来过"，修正累积漂移 → 这是 SLAM 和纯里程计的核心区别

主流方案

  • ▸ ORB-SLAM3 — 视觉 + IMU，学术标杆，代码质量高适合学习原理
  • ▸ RTAB-Map — RGB-D SLAM，ROS2 集成好，适合直接用 → 有 RGB-D 相机的首选
  • ▸ Cartographer — Google 出品，激光 SLAM，2D/3D 均可 → 如果加激光雷达可以考虑
  • ▸ FAST-LIO2 — 激光 + IMU 紧耦合，性能强悍 → 近年来学术界和工业界都很流行

输出：地图

  • ▸ 2D 栅格地图（Occupancy Grid） — 每个格子标记"空闲/占据/未知" → 导航规划最常用的格式
  • ▸ 3D 点云地图 — 稠密的三维环境重建 → 避障更精细但计算量大
这一层解决什么问题：巡航的前提是"有地图"和"知道自己在地图上的位置"。SLAM 解决的就是这两个问题——边走边建图，建完图后还能在这个图上定位。
第五层 · 规划决策
▶ 规划层 Nav2 导航栈 — 从地图到路径 ROS2 自主导航的标准框架

导航流水线

┌─────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ 目标点 │───▶│ 全局规划 │───▶│ 局部规划 │───▶│ 速度指令 │ │(用户指定)│ │(A*/Dijkstra)│ │(DWA/TEB) │ │(cmd_vel) │ └─────────┘ └──────────┘ └──────────┘ └──────────┘ ▲ ▲ │ │ ┌─────────┐ ┌─────────┐ │ 栅格地图 │ │ 传感器 │ │(SLAM输出)│ │(实时避障)│ └─────────┘ └─────────┘
  • ▸ 全局路径规划（Planner） — 在地图上找一条从 A 到 B 的无碰撞路径 → A* 最经典，Nav2 支持多种插件替换
  • ▸ 局部路径规划（Controller） — 实时跟踪全局路径，同时躲避地图上没有的动态障碍 → DWA 在速度空间搜索，TEB 做时间最优
  • ▸ 代价地图（Costmap） — 地图 + 实时传感器数据融合，标记"通行代价" → 障碍物附近代价值高，机器人自然远离
  • ▸ 行为树（Behavior Tree） — 编排复杂导航行为：导航 → 卡住了？→ 旋转清障 → 重试 → Nav2 用 BT 替代了 ROS1 的有限状态机
  • ▸ Recovery Behavior — 导航失败时的恢复策略：旋转、后退、清除代价地图 → 实际部署中非常关键
这一层解决什么问题：建好地图后，"怎么从当前位姿走到目标点"需要回答三个问题——全局路径怎么走（规划）、遇到新障碍怎么办（局部避障）、卡住了怎么办（恢复行为）。Nav2 是 ROS2 社区的标准答案。

第六层 · 运动控制
▶ 控制层 从路径到四足运动 — SDK 的高层控制接口 最后 1 米：让狗走起来

Nav2 输出 → SDK 输入的桥梁

Nav2 输出: cmd_vel (线速度 vx, 角速度 wz) │ ▼ ┌─────────────────────────────────────┐ │ 速度指令转换层 │ │ Nav2 cmd_vel → SDK velocity_sequence│ │ │ │ 关键问题： │ │ · Nav2 默认假设差速驱动机器人 │ │ · 四足的步态 != 轮式运动 │ │ · 需要理解 SDK 状态机流转 │ └─────────────────────────────────────┘ │ ▼ SDK 高层 API (gRPC): RobotClient.state_switch(PASSIVE → STAND → WALK) RobotClient.velocity_sequence(vx, vy, wz)
  • ▸ SDK 状态机 — PASSIVE → STAND → WALK，必须按规则切换 → e3_auto_state_switch.py 展示了完整流程
  • ▸ 速度序列控制 — 给定 vx/vy/wz 驱动行走 → e4_velocity_sequence.py
  • ▸ 路径跟踪控制器 — 把全局路径点序列转换为实时速度指令 → Pure Pursuit（几何跟踪）或 MPC（优化控制）
  • ▸ 步态适配 — 四足不像轮子，转向有惯性、停止有延迟 → 可能需要调参和平滑处理
这一层解决什么问题：Nav2 给出"应该以什么速度走"，SDK 提供了"让四足按速度走"的接口。中间的适配——状态切换、速度平滑、步态约束——是你最终要写的核心代码。
