1.5 ROS2 · SDK-DDS · SDK-gRPC 三者关系
SDK-DDS 和 SDK-gRPC 是两条独立通道（并列，不串联） ROS2 和它们的关系取决于官方交付形式 ════════════════════════════════════════════════════════════════════════════════ SDK-DDS (CycloneDDS) SDK-gRPC (Protobuf + TCP) ┌─────────────────────────┐ ┌─────────────────────────┐ │ 协议: DDS (UDP 多播) │ │ 协议: gRPC (HTTP/2 TCP) │ │ 方向: 机器人 → 外部 │ │ 方向: 外部 → 机器人 │ │ 用途: 传感器数据只读 │ │ 用途: 运动指令控制 │ └────────────┬────────────┘ └────────────┬────────────┘ │ │ │ ROS2 底层通信也是 DDS │ ROS2 和 gRPC 完全无关 │ 同一个协议！ │ 两套独立系统，互不认识 │ │ ▼ ▼ ┌──────────────────────────────────────────────────────────────────┐ │ ROS2 的通信层 (rmw) │ │ │ │ ROS2 不发明新的通信协议，它在 DDS 上面加了一层封装： │ │ │ │ ROS2 Node ──→ rclcpp ──→ rmw ──→ CycloneDDS ──→ UDP 网络传输 │ │ ↑ ↑ │ │ │ 这里和 SDK-DDS │ │ │ 用的是同一个 DDS │ │ │ │ 所以：如果 ROS2 配置为 CycloneDDS（和你 SDK 一样）， │ │ ROS2 节点理论上可以直接订阅 SDK 的 DDS Topic，不需要翻译。 │ │ 但消息格式（IDL 自定义类型 vs sensor_msgs）可能对不上。 │ └──────────────────────────────────────────────────────────────────┘ ════════════════════════════════════════════════════════════════════════════════ 关系总结： SDK-DDS ←─ 同协议，天然亲近 ─→ ROS2 ↖ ↗ 完全无关，互不认识 ↙ ↘ SDK-gRPC ROS2
▶ 关系 ROS2 和 SDK-DDS：同协议，但消息格式不同
  • ▸ 协议层面：ROS2 底层可以用 CycloneDDS，和 SDK 用的是同一个。UDP 多播发现、Topic 发布订阅机制完全一样
  • ▸ 消息格式：SDK 的 DDS Topic 用自定义 IDL 定义（越疆私有的图像/IMU 格式），ROS2 用标准 sensor_msgs/Image、sensor_msgs/Imu。格式不同，不能直接互通
  • ▸ 如果要互通：需要写一个桥接节点，订阅 DDS Topic → 反序列化 → 重新封装为 ROS2 消息发布。这就是之前方案里 Bridge A 做的事
  • ▸ 你现在的方案：等官方提供 SLAM+导航，如果官方在 RK3588 上直接封装好 gRPC 接口，你 PC 上不需要碰 DDS，也不需要 ROS2
▶ 关系 ROS2 和 SDK-gRPC：完全无关
  • ▸ gRPC 是独立 RPC 框架，不走 DDS，和 ROS2 没有任何交集
  • ▸ ROS2 的 Service/Action 也能实现请求-响应，但底层走的是 DDS，不是 gRPC
  • ▸ 如果要把 ROS2 的导航指令发给 gRPC：需要写桥接节点，订阅 ROS2 /cmd_vel → 调用 gRPC velocity_sequence()。这就是之前方案里 Bridge B 做的事
  • ▸ 你现在的方案：PC 程序直接调 SDK gRPC Client，不经过 ROS2。所以 Bridge B 也不需要
▶ 决策 你现在的方案里，ROS2 在哪？
如果官方以 gRPC API 形式交付（最可能）： 你的 PC 程序 ──gRPC──▶ RK3588 (SLAM+Nav+小脑) ──DDS──▶ (可选监控传感器) ROS2 = 完全不参与。你的程序直接调 SDK，不经过 ROS2。 ────────────────────────────────────────────────────────── 如果官方以 ROS2 包形式交付： 你的 PC 程序 ──ROS2 Topic──▶ ROS2 节点 ──DDS──▶ RK3588 ──ROS2 Service──▶ ROS2 节点 ──gRPC──▶ RK3588 ROS2 = 夹在你的程序和 SDK 之间，作为中间层。 你需要：在 PC 上装 ROS2 + 编译官方 ROS2 包 + 启动节点。
结论：在确认官方交付形式之前，你的 PC 程序架构按"直接调 SDK"来设计。如果官方给的是 ROS2 包，加一层 ROS2 适配就行；如果是 gRPC API，现在的架构已经对了。

7 硬件约束与边界
资源规格意味着什么
RK3588 CPU 8 核 ARM (4×A76 + 4×A55) 跑 Linux + SLAM + Nav2 够用，但不能上重算法
内存 8 GB 2D 激光 SLAM 足够，3D 点云建图会比较紧张
NPU 算力 6 TOPS 可跑轻量 AI 推理（目标检测等），不够跑大模型
主传感器 360° 激光雷达 适合 2D 栅格建图，不支持 3D 环境重建
辅助传感器 D435i 深度相机 RGB-D 图像，用于局部避障和短距离感知
网络 有线 192.168.5.2 + WiFi DDS 必须有线；gRPC 走 WiFi 也行
可选扩展 额外算力背包 如果 8GB 不够，可外接算力模块
8 风险与待确认事项
▶ 待确认 需要问清楚官方的问题
  • 1. 交付形式 — SDK 新增 gRPC 接口（调 API），还是 ROS2 功能包（需要配 ROS2 环境）？
  • 2. 建图精度 — 激光雷达的实际建图质量如何？能否应对玻璃门、长走廊等退化场景？
  • 3. 开放程度 — 是黑盒 API（只能调 navigate_to），还是能拿到地图数据、自定义规划策略？
  • 4. 多层管理 — 多楼层地图如何切换？机器人怎么知道自己换了楼层？
  • 5. 时间表 — 6 月底是确定日期还是估计？会不会延期？
  • 6. 算力占用 — SLAM + 导航跑起来后，占多少内存/CPU？是否影响小脑的实时性？
▶ 风险 可能卡住的地方
  • ▸ 官方延期 — 6 月底可能不是确定日期。提前准备：如果延期，是否需要临时方案？
  • ▸ 功能不满足需求 — 官方导航可能只有基础功能（单目标导航），不包含多目标巡航、自动充电回桩等。评估差距
  • ▸ 8GB 内存瓶颈 — SLAM + 导航 + 小脑同时运行，如果内存吃紧可能导致卡顿或崩溃
  • ▸ 2D 激光局限 — 纯 2D 建图对透明障碍物（玻璃门）、低矮障碍物（台阶边缘）可能检测不到
9 参考项目：同行的做法

▶ 参考 宇树 GO2 自主导航开源项目
  • ▸ 宇树的优势 — GO2 底层 DDS 天然兼容 ROS2（同协议），不需要写桥接层；越疆多了 gRPC，需要额外翻译
